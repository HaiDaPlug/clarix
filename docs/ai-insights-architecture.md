# AI Insights Architecture

## The problem we're solving

The report and dashboard currently have two problems:

1. **Static copy** — headlines like "Din digitala synlighet går åt rätt håll" are hardcoded and don't reflect what the data actually says for that client in that period.
2. **Template feel** — `NextStepsCard`, `SlideStrategicInsight`, and `SlideRecommendations` use rule-based strings that read like a dashboard, not an advisor.

The goal: every client opens their report and it reads like it was written for them, based on their actual data.

---

## Core principle

**Deterministic where it's a number or a label. LLM where it's a sentence that should feel like an advisor wrote it.**

Numbers, percentages, trend pills, channel names — all derived from data, never touched by AI.

Sentences that synthesize, educate, or advise — generated by Claude, stored at cache time, served instantly.

---

## Architecture: three layers

### Layer 1 — Data (already built)
Raw metrics from GA4 and Search Console, normalized into `ReportData`. `buildSlideData()` in `page.tsx` handles this. No changes needed here.

### Layer 2 — Insight classifier (to build)
A pure TypeScript function `deriveInsights(data: ReportData): Insight[]` that takes the normalized data and produces structured insight objects — no copy, no LLM, just classified facts.

```ts
interface Insight {
  type: InsightType;
  severity: "positive" | "neutral" | "warning" | "critical";
  metrics: Record<string, number | string | null>;  // real values only
  surface: InsightSurface[];  // which UI slots consume this
}

type InsightType =
  | "traffic_up_broadly"
  | "traffic_down_broadly"
  | "traffic_drop_organic"
  | "traffic_drop_paid"
  | "traffic_channel_concentrated"
  | "engagement_down"
  | "engagement_up"
  | "contact_page_lost_visibility"
  | "paid_roas_strong"
  | "paid_cost_up_conversions_flat"
  | "seo_positions_improving"
  | "seo_positions_declining"
  | "conversion_rate_improved"
  | "conversion_rate_declined"
  | "ai_visibility_untracked"
  | "data_missing_tracking_issue";

type InsightSurface =
  | "dashboard_hero"
  | "next_steps"
  | "slide_hero"
  | "slide_insight"
  | "slide_recs"
  | "slide_recap";
```

Severity thresholds:
- `trafficDelta < -20%` → critical
- `trafficDelta < -10%` → warning
- `trafficDelta between -5% and +5%` → neutral
- `trafficDelta > +10%` → positive

This layer is the editorial brain of Clarix. It is never random and never drifts.

### Layer 3 — LLM generation (to build)
One API route (`/api/generate-insights`) receives the classified `Insight[]` array plus scoped metric values and calls Claude with a structured prompt. Claude returns a JSON object with copy for all 6 surfaces in one shot.

---

## Prompt anatomy — role before text

Before Claude generates a single word for any surface, it is given three things in order:

**1. ROLE** — who Claude is on this surface and what job it is doing.
This changes per surface. A pedagogical role explains what a metric means for a non-technical SME owner. An advisory role identifies what the client should do next. These are fundamentally different tones and must not bleed into each other.

**2. DATA** — only the metrics this surface is responsible for, nothing else.
`slide_hero` gets traffic delta and top channel. `slide_insight` gets the full picture. Scoping the data prevents the LLM from referencing numbers the user hasn't seen on that slide yet.

**3. CONSTRAINT** — the output format and length, hard-coded per surface.
"2 sentences max", "3 paragraphs", "one concrete action verb per step". This is what keeps the output from drifting.

### Role taxonomy

| Surface | Role | Type | Tone |
|---|---|---|---|
| `slide_hero` | Summarizer | Educational — what happened this period | Calm, factual |
| `slide_insight` | Strategist | Analytical — what this means for the business | Incisive, forward-looking |
| `slide_recs` | Advisor | Actionable — what to do next and why | Direct, confident |
| `slide_recap` | Editor | Distillation — three things to remember | Crisp, memorable |
| `dashboard_hero` | Narrator | Educational — one-line read on the period | Warm, accessible |
| `next_steps` | Coach | Actionable — why this specific step for this client | Motivating, grounded |

### Data scoping per surface

| Surface | Gets this data |
|---|---|
| `slide_hero` | `visits`, `visitsDelta`, `topChannel`, `topChannelPct` |
| `slide_insight` | Full insight array, all metrics — this is the synthesis slide |
| `slide_recs` | The 3 derived `Insight` objects with severity and metrics |
| `slide_recap` | Top 3 insights by severity |
| `dashboard_hero` | `visits`, `visitsDelta`, `trend` |
| `next_steps` | Rule-derived action label + the metric that triggered it |

The prompt builder is `buildPromptForSurface(surface, insights, metrics)` — one function, six branches. Claude receives all six in a single call, each section scoped separately inside the prompt.

---

## Null state — insufficient data

If the classifier determines there is not enough data for a surface to generate a meaningful insight, it returns `null` for that slot instead of generating thin or speculative copy.

**Threshold for null:** a surface returns `null` if the primary metrics it depends on are all absent or zero with no previous period to compare against.

Claude is instructed: *"If the data for a surface is marked as insufficient, return `null` for that key."*

The UI fallback for any `null` slot:

```
Inte nog med data för att bedöma din digitala närvaro.
```

This appears in place of the generated copy, styled the same way. Never an error state, never a broken card — just an honest placeholder that respects the client.

---

## The 6 copy slots

| Surface | Component | What gets generated | Null condition |
|---|---|---|---|
| `dashboard_hero` | `DashboardHero.tsx` | Headline + 1-sentence subheadline | No traffic data at all |
| `next_steps` | `NextStepsCard.tsx` | Rationale per step (action label stays rule-based) | Fewer than 2 data sources connected |
| `slide_hero` | `SlideHero` in `page.tsx` | AISummary paragraph(s) on slide 1 | No sessions data |
| `slide_insight` | `SlideStrategicInsight` in `page.tsx` | 3 body paragraphs + bottom line | No previous period to compare |
| `slide_recs` | `SlideRecommendations` in `page.tsx` | Body text per recommendation card | No classifier insights with severity ≥ warning |
| `slide_recap` | `SlideRecap` in `page.tsx` | Body line per "tre saker att ta med dig" item | No classifier insights at all |

The report slide headlines (e.g. "Din digitala synlighet går åt rätt håll") vary via a deterministic lookup table keyed on `InsightType` — not LLM. `traffic_down_broadly` renders a different headline than `traffic_up_broadly`.

## `AiInsightsPayload` — the stored shape

```ts
interface AiInsightsPayload {
  dashboard_hero: {
    headline: string;
    sub: string;
  } | null;
  next_steps: ({
    rationale: string;
  } | null)[];
  slide_hero: string | null;
  slide_insight: {
    body: string[];        // 3 paragraphs
    bottom_line: string;
  } | null;
  slide_recs: ({
    body: string;
  } | null)[];
  slide_recap: ({
    body: string;
  } | null)[];
}
```

Every slot is nullable. The UI always checks for null before rendering AI copy.

## Caching

### Table: `ai_report_cache`

```sql
create table ai_report_cache (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references auth.users not null,
  period_start date not null,
  period_end date not null,
  metrics_hash text not null,
  generated_at timestamptz not null default now(),
  insights jsonb not null,
  unique (user_id, period_start, period_end)
);
```

`metrics_hash` is a short hash of the key input values (visits, topChannel, bounceRate, leads). If GA4 backfills data and the hash changes, the cache is invalidated and regeneration fires on next dashboard load.

### Invalidation rules

Regenerate if:
- Period changes (`period_start` / `period_end` differ from cached row)
- `metrics_hash` differs from the cached row
- Cache row is older than 24 hours (safety net)

Otherwise serve from cache. Opening the report is always instant.

---

## Trigger logic

**The dashboard load generates. The report only reads.**

On dashboard load:
1. Check `ai_report_cache` for `(user_id, current_period_start, current_period_end)`
2. **Cache hit + valid hash** → serve stored insights immediately
3. **Cache miss or stale** → serve deterministic fallbacks immediately, fire `/api/generate-insights` in the background, write to cache

Generation takes ~2–4 seconds. A lightweight client-side refetch after 5 seconds picks up the result. The user never waits.

### Why not a cron?
Adds complexity before it's needed. Runs for all users regardless of whether they open the dashboard. Add later for pre-warming once the core is proven.

### Why not generate on report open?
Report must open instantly. Dashboard is always the natural entry point first. One source of truth, no race conditions.

## Generation API route

`POST /api/generate-insights`

**Input:**
```ts
{
  userId: string;
  period: { start: string; end: string; label: string };
  insights: Insight[];
  metrics: {
    visits: number;
    visitsDelta: number | null;
    topChannel: string;
    topChannelPct: number;
    bounceRate: number | null;
    avgDuration: number | null;
    leads: number;
  };
}
```

**Flow:**
1. Check cache — return early if valid
2. Run classifier to get `Insight[]`
3. For each surface, check null threshold — mark insufficient surfaces
4. Build prompt with role + scoped data + constraint per surface
5. Call `claude-sonnet-4-6`, `max_tokens: 1200`
6. Parse response as `AiInsightsPayload` (null slots stay null)
7. Compute `metrics_hash`, upsert to `ai_report_cache`
8. Return payload

**Error handling:** if generation fails, return `{ fallback: true }`. UI keeps deterministic copy. Never block on LLM failure.

## What stays deterministic forever

- All numbers, percentages, visit counts
- Trend pill colors and direction
- Channel names and icons
- Slide headlines (lookup table by `InsightType`)
- KPI card labels
- Page URLs in "most visited pages"
- Anything in `buildSlideData()`

## Build order

1. **Insight classifier** — `src/lib/engine/derive-insights.ts`
2. **Deterministic slide headlines** — lookup table in `SlideHero` keyed on `InsightType`
3. **Supabase migration** — `ai_report_cache` table
4. **Generation API route** — `src/app/api/generate-insights/route.ts`
5. **Dashboard integration** — `DashboardHero` and `NextStepsCard` read from cache, trigger on miss
6. **Report integration** — `SlideHero`, `SlideStrategicInsight`, `SlideRecommendations`, `SlideRecap` read from cache, null → fallback string

---

## Prompt design principles

- Role defined before data, data defined before constraint — in that order, every surface
- Send structured facts, not raw data dumps
- Enumerate allowed recommendation types so Claude doesn't invent actions we can't support
- Specify exact JSON output shape in system prompt
- Tone: "calm, pedagogical Swedish business advisor — never hype, never vague"
- Hard constraint: "use only the facts in the input, do not speculate"
- One call, all 6 surfaces — cheaper and more coherent than 6 separate calls
- Null instruction: "if data for a surface is marked insufficient, return null for that key"
